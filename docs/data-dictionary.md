# Data Dictionary

This file describes the practical data contracts a CAMA/vendor engineer should look for first. It is intentionally product-facing rather than exhaustive.

## Property Manifest

Path: `data/app/property-manifest.json`

| Field | Meaning |
| --- | --- |
| `activePropertyId` | Property id loaded by the demo. |
| `properties[]` | Demo property records available to the app. |
| `properties[].recordCardPath` | Path to the active vendor/source property record card. |
| `sharedData.counties` | County-scoped datasets keyed by county slug. |
| `sharedData.*Path` | Static app-ready reference data paths. |

## Vendor Property Record

Path: `data/property-records/mips/residential-010496000-record-card.json`

Adapter notes: `src/adapters/mips/record-card.js` extracts the current demo `guidedSnapshot`, and `src/adapters/mips/field-map.json` groups the source fields that need a future canonical adapter.

| Field | Meaning |
| --- | --- |
| `source` | Source system, report/card metadata, printed date, and notes. |
| `guidedSnapshot` | Current app-ready property snapshot. This should eventually be generated from canonical raw property data. |
| `parcelIdentifiers` | Vendor and jurisdiction identifiers. |
| `locationModel` | Neighborhood, district, valuation group, model, and method. |
| `landModel` | Land model, lot size, recorded lot value, and cutoff schedule. |
| `currentCardValue` | Prior, initial current, and reconciled current value breakdowns. |
| `valuationHistory` | Record-card value/tax history by statement year. |
| `propertyValuation` | Current recorded valuation rollup. |
| `valuationReconciliation` | Notes and bridge values when source card values differ from final assessed values. |
| `residentialInformation` | Vendor-style dwelling facts used by the cost model. |
| `costApproach` | Cost model inputs and RCNLD/depreciation outputs. |
| `garageCostLines` | Garage valuation components. |
| `miscImprovements` | Other site/improvement valuation components. |

## Guided Snapshot

Current location: `recordCard.guidedSnapshot`

Target role: app-ready property snapshot generated by an adapter/model layer.

| Field | Meaning |
| --- | --- |
| `snapshotYear` | Current assessment year shown by the app. |
| `latestFinalTaxYear` | Most recent final tax statement year. |
| `assets` | Local demo image/sketch paths. |
| `parcel` | Taxpayer-facing parcel identity. |
| `classification` | Class, status, location, zoning, lot label. |
| `residential` | Normalized residential facts shown in the UI. |
| `taxpayerHistory` | High-level year/value/tax/status rows for charts. |
| `taxStatements` | Statement-level tax rows with credits and net due. |
| `districtLevyHistory` | Consolidated levy history and status notes. |
| `assessedValueBreakdown` | Land/dwelling/outbuilding/total rows by year. |
| `latestFinalLevyComponents` | Latest final taxing-body levy components. |
| `dwellingData`, `outbuildingData`, `landInformation` | App-ready property detail rows. |
| `sources`, `sourcePolicy` | UI source notes and canonical-record policy. |

## County/State Reference Data

| Dataset | Primary Use |
| --- | --- |
| `data/statewide/certified-taxes-levied.json` | County/state CTL value, taxes, and average-rate comparisons. |
| `data/statewide/county-ctl-comparisons.json` | Normalized CTL rows with source verification fields. |
| `data/statewide/pad-ratio-statistics-by-county.json` | County/class ratio-study comparison rows. |
| `data/counties/gage/assessment-ratio-analysis.json` | Gage historical equalization context. |
| `data/counties/gage/market-position-statistics-2026-gage.json` | Class-aware market-area scatter/card data. |
| `data/counties/gage/tax-district-authorities-2025.json` | Tax district authority list and levy rollup. |
| `data/sources/nebraska-pad-source-registry.json` | Source-document registry. |
| `data/sources/nebraska-pad-metric-ledger.json` | Deep provenance and extraction ledger; not intended for eager browser loading. |

## Naming Rules

- Use `taxYear` for tax statement and CTL years.
- Use `assessmentYear` for assessment/ratio-study values.
- Use `reportYear` for report publication year.
- Use `statementYear` only for statement/calendar contexts.
- Keep source values and normalized/app values separate when units differ.
